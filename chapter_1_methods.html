2  Methods and Validation

2.1 Pipeline overview

The extraction pipeline is a linear LangGraph workflow with eight deterministic nodes. No unconstrained agents; the LLM is used only for extraction and normalization.

Key design decisions:

  • The LLM assigns candidate_sdgs, evidence_tags, implementation_stage, and confidence — all normalization tasks
  • Validation, scoring, and final strength assessment are deterministic Python — the model cannot inflate ratings
  • Conservative extraction: when uncertain, the model is instructed to return no evidence rather than a speculative item

2.2 Scoring rubric

The computed_score (0–13) is calculated from four additive components. The model is never asked to assign weak/moderate/strong directly.

Evidence scoring rubric (documented max = 13)
Component Points Notes
A. Action maturity — mention_only 1 SDG is mentioned without any action language
A. Action maturity — planned_action 2 Stated plans or commitments exist
A. Action maturity — implementation_in_progress 3 Active implementation underway
A. Action maturity — implemented_with_measurable_evidence 4 Outcomes are measured and reported
B. Tag bonus — policy 1 +1 if evidence references a policy
B. Tag bonus — initiative 1 +1 if a specific initiative is named
B. Tag bonus — target 1 +1 if a target is stated
B. Tag bonus — kpi 1 +1 if a KPI is tracked
B. Tag bonus — measured_outcome 2 +2 if a measured outcome is present (e.g., numbers, percentages)
C. Support — quantitative_support 1 +1 if numeric or metric-like evidence is present
C. Support — oversight_support 1 +1 if board, committee, or executive oversight is explicitly named
Strength: 1–2 = weak · 3–5 = moderate · 6+ = strong

2.3 Validation rules

Eight deterministic rules must all pass for an evidence item to be retained. Failure causes rejection with the specific rule recorded.

2.4 Extraction quality audit

For each report: how many chunks were retrieved, how many evidence items extracted, how many passed validation.

2.5 Construct validity checks

2.5.1 1. Does model confidence correlate with computed score?

Higher model-reported confidence should correlate with higher deterministic scores if the model’s self-assessment is calibrated.

2.5.2 2. Tag–stage coherence

Do higher-maturity tags co-occur with higher implementation stages? A measured_outcome tag with a mention_only stage would be incoherent.

2.6 Known limitations

Warning

Retrieval ceiling: top_k = 20 chunks per report. Evidence in unchosen chunks is not extracted. Raising top_k increases recall but also extraction time (approximately 4 minutes per chunk on CPU with Gemma2).

Early reports (2016–2019): Shorter, less structured PDFs. Section headings were less consistently detected. Comparisons across the full decade should account for this.

Single model: All reports were extracted with Gemma2 (Ollama). No inter-model reliability check has been performed. Model comparison is deferred to Phase 2.

Chunk boundary effects: Evidence spanning two chunks may be split. The chunker uses 80-token overlap to mitigate this, but some cross-boundary items may be missed or duplicated.
